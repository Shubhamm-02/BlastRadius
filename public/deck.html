The problem

“If I change this function… what breaks?”

  • Every developer asks this before touching shared code.
  • Your IDE's “Find References” shows only direct callers - one hop.
  • The real danger is transitive: the function four calls away, in a flow you forgot about.
  • Today's answer: grep, guesswork, and hoping the tests catch it.
The insight

Code is a graph.
Impact is a traversal.

Functions call functions - that's a directed graph. “What breaks?” is simply everything upstream that can reach this node.

A graph question deserves a graph database.

What it does

Point it at a repo. Click a function. See the blast radius.

1. Paste a GitHub URL - or upload a local folder.

2. We build a call graph in Neo4j AuraDB.

3. Click any function → its full blast radius lights up.

4. Sarvam AI explains what to double-check before shipping.

Why a graph database

The query that a relational DB can't do cleanly

MATCH (target:Function {id: $id})<-[:CALLS*1..6]-(affected:Function)
RETURN DISTINCT affected
  • “Everything up to 6 calls away” - one line of Cypher.
  • The same thing in SQL = recursive joins / CTE gymnastics.
  • This variable-length traversal is the graph-native core. Neo4j AuraDB does the heavy lifting.

The data model

A small, powerful schema

(:File {path})
(:Function {id, name, file, line})

(:File) -[:IMPORTS]-> (:File)
(:Function) -[:DEFINED_IN]-> (:File)
(:Function) -[:CALLS]-> (:Function)

CALLS is the star. Every project is tagged with a projectId, so many codebases live in one database without colliding.

What made it hard

Getting the graph right

  • Cross-file calls resolved to the import, not the function - we followed the alias to the real declaration. (edges 4 → 14 on our test project)
  • Path aliases (@/…) broke resolution on modern repos - we read tsconfig to fix it. (imports 8 → 17, calls 14 → 28)
  • Big graphs are unreadable - we added search + a focused blast-radius subgraph so it scales.
